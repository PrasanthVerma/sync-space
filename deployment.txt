================================================================================
  3. CRITICAL PRE-DEPLOYMENT FIXES REQUIRED
================================================================================

ISSUE 1 — Execution Engine: Docker-in-Docker NOT configured
  Current runner.js runs plain `exec(runCommand)` WITHOUT spinning Docker
  containers. The Dockerfiles exist but are not used in runner.js.
  The runner must use:
    docker run --rm --memory="128m" --cpus="0.5" --network none \
      -v <tempDir>:/app \
      syncspace-runner-node:latest node script.js
  Any production host for the execution-engine MUST have Docker daemon
  access (privileged mode or a VM — NOT a standard container platform).

ISSUE 2 — CORS: server/src/index.js uses app.use(cors()) with no origin
  In production, restrict to your frontend domain:
    app.use(cors({ origin: 'https://<frontend-domain>', credentials: true }));

ISSUE 3 — Yjs in-memory state lost on server restart
  y-websocket stores documents in RAM only. If server restarts, all active
  sessions lose their document state. Fix: use y-leveldb persistence layer
  or store snapshots to MongoDB on room save.

ISSUE 4 — No rate limiting on auth routes
  /api/auth/register and /api/auth/login have no brute-force protection.
  Add express-rate-limit before deploying publicly.

ISSUE 5 — No Dockerfile for server or client
  All three services need Dockerfiles for containerised deployment.

  4. DEPLOYMENT OPTIONS
================================================================================

────────────────────────────────────────────────────────────────────────────────
  OPTION A — Recommended: Vercel (client) + Render (server + exec-engine)
────────────────────────────────────────────────────────────────────────────────

This is the simplest path for a solo developer / small team.
MongoDB Atlas is already configured in your .env — keep it.

A1. CLIENT → Vercel
    ─────────────────
    1. Push repo to GitHub.
    2. Go to vercel.com → New Project → Import your repo.
    3. Set Root Directory: client/
    4. Build Command: npm run build
    5. Output Directory: dist
    6. Add Environment Variables in Vercel dashboard:
         VITE_API_URL       = https://syncspace-server.onrender.com
         VITE_WS_URL        = wss://syncspace-server.onrender.com
         VITE_EXEC_URL      = https://syncspace-exec.onrender.com
    7. Deploy. Vercel serves the SPA globally via CDN.
    8. Add a vercel.json to handle SPA routing:

       {
         "rewrites": [{ "source": "/(.*)", "destination": "/index.html" }]
       }

    Cost: Free tier (100GB bandwidth/month)

A2. SERVER (Backend) → Render Web Service
    ─────────────────────────────────────────
    1. In Render dashboard → New → Web Service.
    2. Connect GitHub repo, set Root Directory: server/
    3. Build Command: npm install
    4. Start Command: npm start          (runs: node src/index.js)
    5. Plan: Starter ($7/mo) — free tier sleeps after 15min inactivity,
       which breaks persistent WebSocket connections. Use Starter minimum.
    6. Add Environment Variables:
         MONGO_URI   = <atlas-uri>
         PORT        = 5000
         JWT_SECRET  = <generated-secret>
         REDIS_URL   = redis://<upstash-host>:6379
    7. Enable "Auto-Deploy" on push to main branch.

    WEBSOCKET NOTE: Render supports WebSocket upgrades natively on
    its Starter plan and above. No extra config needed.

A3. REDIS → Upstash (Serverless Redis)
    ───────────────────────────────────
    1. Go to upstash.com → Create Database → Redis.
    2. Select region closest to your Render server.
    3. Copy the Redis URL (format: rediss://... with TLS).
    4. Set REDIS_URL in Render environment.
    Cost: Free tier (10,000 commands/day) is sufficient for token blacklisting.

A4. EXECUTION ENGINE → Render (requires Docker)
    ──────────────────────────────────────────────
    IMPORTANT: The execution engine spawns Docker containers.
    Standard Render Web Services run inside containers and do NOT have
    Docker daemon access. Options:

    Option A4a — Render with Docker-in-Docker (not supported on Render):
      NOT viable on Render's standard offering.

    Option A4b — Run on a VPS (DigitalOcean Droplet / Linode / Hetzner):
      1. Provision a $6/mo DigitalOcean Droplet (Ubuntu 22.04, 1GB RAM).
      2. SSH in and run:
           apt update && apt install -y nodejs npm docker.io
           systemctl enable --now docker
      3. Clone repo, cd execution-engine, npm install
      4. Build sandbox images:
           docker build -t syncspace-runner-node:latest -f Dockerfile.node .
           docker build -t syncspace-runner-python:latest -f Dockerfile.python .
      5. Fix runner.js to use docker run (see Section 5 below).
      6. Run with PM2 for process management:
           npm install -g pm2
           pm2 start src/index.js --name syncspace-exec
           pm2 save && pm2 startup
      7. Use Nginx as a reverse proxy with Let's Encrypt SSL:
           apt install -y nginx certbot python3-certbot-nginx
           certbot --nginx -d exec.yourdomain.com
      8. Point VITE_EXEC_URL to https://exec.yourdomain.com

    Option A4c — Disable real Docker execution (simplest workaround):
      Replace Docker execution with a sandboxed vm2 or quickjs-emscripten
      for JavaScript only. Python support would be dropped.
      This eliminates the Docker requirement entirely.

B3. Nginx reverse proxy config (nginx/nginx.conf)

events {}
http {
  upstream backend  { server server:5000; }
  upstream execeng  { server execution-engine:5001; }

  server {
    listen 80;
    server_name yourdomain.com;
    return 301 https://$host$request_uri;
  }

  server {
    listen 443 ssl;
    server_name yourdomain.com;
    ssl_certificate     /etc/letsencrypt/live/yourdomain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/yourdomain.com/privkey.pem;

    # Frontend (SPA)
    location / {
      proxy_pass http://client:80;
    }

    # Backend API + WebSocket
    location /api/ {
      proxy_pass http://backend;
      proxy_http_version 1.1;
    }
    location /yjs/ {
      proxy_pass http://backend;
      proxy_http_version 1.1;
      proxy_set_header Upgrade $http_upgrade;
      proxy_set_header Connection "Upgrade";
      proxy_set_header Host $host;
      proxy_read_timeout 86400;
    }

    # Execution Engine
    location /exec/ {
      rewrite ^/exec/(.*)$ /$1 break;
      proxy_pass http://execeng;
    }
  }
}

================================================================================
  5. RUNNER.JS — REQUIRED FIX FOR DOCKER EXECUTION
================================================================================

The current runner.js (execution-engine/src/runner.js) does NOT use Docker.
Replace the exec() call with a docker run command:

--- REPLACE exec(runCommand, ...) BLOCK WITH ---

    const dockerCmd = [
      'docker run',
      '--rm',
      '--network none',
      '--memory=128m',
      '--cpus=0.5',
      '--pids-limit 64',
      '--read-only',
      `--tmpfs /app:rw,size=10m`,
      `-v ${filePath}:/app/${fileName}:ro`,
      `-w /app`,
      dockerImage,
      runCommand
    ].join(' ');

    exec(dockerCmd, { timeout: 10000 }, async (error, stdout, stderr) => {
      // cleanup and resolve as before
    });

Security flags explained:
  --network none     : No internet access from sandbox
  --memory=128m      : Cap memory at 128MB
  --cpus=0.5         : Cap CPU usage
  --pids-limit 64    : Prevent fork bombs
  --read-only        : Container filesystem is read-only
  --tmpfs /app       : Writable temp dir in RAM only (wiped on exit)
  -v <file>:ro       : Mount only the script, read-only

  6. CI/CD PIPELINE (GitHub Actions)
================================================================================

Create .github/workflows/deploy.yml:

name: CI/CD

on:
  push:
    branches: [main]

jobs:
  lint-and-build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install & Build Client
        working-directory: client
        run: |
          npm ci
          npm run build
        env:
          VITE_API_URL: ${{ secrets.VITE_API_URL }}
          VITE_WS_URL: ${{ secrets.VITE_WS_URL }}
          VITE_EXEC_URL: ${{ secrets.VITE_EXEC_URL }}

      - name: Lint Client
        working-directory: client
        run: npm run lint

  deploy-server:
    needs: lint-and-build
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - name: Trigger Render Deploy (server)
        run: curl -X POST ${{ secrets.RENDER_SERVER_DEPLOY_HOOK }}

  deploy-exec-engine:
    needs: lint-and-build
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - name: SSH and redeploy on VPS
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.VPS_HOST }}
          username: ${{ secrets.VPS_USER }}
          key: ${{ secrets.VPS_SSH_KEY }}
          script: |
            cd /opt/syncspace
            git pull origin main
            docker compose up -d --build execution-engine

Required GitHub Secrets:
  VITE_API_URL, VITE_WS_URL, VITE_EXEC_URL
  RENDER_SERVER_DEPLOY_HOOK  (from Render dashboard → Settings → Deploy Hook)
  VPS_HOST, VPS_USER, VPS_SSH_KEY

  7. MONGODB ATLAS CONFIGURATION
================================================================================

Your MONGO_URI already points to Atlas. In production:

1. Go to Atlas → Network Access → Add IP Address.
   - For Render: Add Render's outbound IPs (listed in Render docs) OR
     use 0.0.0.0/0 temporarily and restrict later.
   - For VPS: Add your VPS static IP only.

2. Go to Atlas → Database Access → Edit user "Prasanth":
   - Rotate the password after seeing it in plain text in .env.
   - Set built-in role to: readWrite on database "syncspace" ONLY.

3. Enable Atlas Backup:
   - Go to your cluster → Backup → Enable.
   - Schedule daily snapshots.

4. Set Atlas Alerts:
   - Alert if connections > 80% of limit.
   - Alert if disk usage > 70%.

5. Index recommendations (add via Atlas or Mongoose):
   User.email   — unique index (already in schema, ensure it's in Atlas)
   Room.roomId  — unique index (already in schema)

  Step 19. Import repo to Vercel. Set root directory to client/.
  Step 20. Add VITE_* environment variables pointing to deployed services.
  Step 21. Add vercel.json with SPA rewrite rule.
  Step 22. Deploy and verify:
             - Login / Register flows work.
             - Create room → editor loads.
             - Monaco editor syncs in real-time across two browser tabs.
             - Code execution returns output.

PHASE 6 — Smoke Tests
  Step 23. Open app in two incognito windows. Join same room.
           Verify keystrokes sync in real-time.

  Step 25. Run Python code → verify output appears.
